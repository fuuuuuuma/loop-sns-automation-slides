CHAPTER 1

ループエンジニアリングとは
定義

「ゴール」と「検証方法」を設計して、
AIが自分で回り続けるループを作る技術

人間が毎回プロンプトを打つのではなく、
「何を達成したら終わりか」「できたかどうかをどう確かめるか」を先に決めて、あとはAIに任せる。

プロンプトの時代は終わった

これまで

プロンプト
エンジニアリング ＝時代遅れ

  • 1回の指示 → 1個の成果物
  • 毎回、人間が考えて打つ
  • AIが終わったら止まる
これから

ループ
エンジニアリング ＝新常識

  • 1回の設計 → 成果物が出続ける
  • 次の指示はループが考える
  • 終わっても自分で再開する

CHAPTER 2

「AI社員」時代の終わり
少し前までの正解

「AI社員を雇おう」
「AIに役割を与えて指示を出そう」

ライター係・リサーチ係・編集係……
AIをスタッフのように使うのが最先端と言われてきました。

でも、この方式には致命的な欠陥があります

致命的な欠陥：人間がボトルネック

AI社員は優秀。でも、ずっと「指示待ち」

  • 指示を出せるのは、人間が PC の前にいる時間だけ
  • 納品されたら、次の指示をまた人間が考える
  • AI社員が増えるほど、管理する人間の仕事が増える

数字で見ると、もったいなさが分かる

AIは 24時間365日 働けるのに、
人間が指示を出せるのは 1日ほんの数回

指示待ちの時間 ＝ AIが止まっている時間。
性能ではなく「指示の供給」が上限になっている。

AI社員方式 vs ループ方式

AI社員方式

指示するたびに動く

  • 指示 → 納品 → また指示
  • 夜は止まる
  • 社員が増えると管理が増える
ループ方式

設計したら動き続ける

  • 設計1回 → あとは自走
  • 寝ている間も回る
  • 次の指示はループが出す
誤解しないでほしいこと

AI社員が消えるわけではありません。
「雇い主」が人間からループに変わるんです

ループが必要なタイミングで、必要な数のサブエージェント（＝AI社員）を自動で雇い、
仕事が終わったら自動で解散する。人間は採用面接すらしなくていい。

AI社員を雇って、指示する時代は
もう終わりました。

永久機関の設計図

素材：YouTube動画 1本（台本・文字起こし）

/loop が各媒体向けに 変換 → 検証 → 投稿

X 投稿文 note 記事 Tips 記事 Threads 連投 リール台本 TikTok台本 ショート字幕

動画1本 → 7媒体分のコンテンツに自動分解。これが1ループ分

1周のループでやっていること

  1. 素材から、媒体ごとのコンテンツを生成
    X用・note用・Tips用…それぞれ専用スキルが文体を変える
  2. フォーマットとファクトを検証
    文字数・構成・固有名詞・URL を自分でチェック
  3. 投稿 or 下書き保存
    媒体ごとに「どこまで自動でやるか」を決めておく
  4. 結果を記録して、次の素材へ
    失敗パターンは memory に追記 → ループは回るほど賢くなる

人間が寝ている約8時間 ＝ これまで「AIが止まっていた」時間が、まるごと生産時間に変わる

人間の仕事

残る仕事は、たった2つ

① 最初のループ設計
ゴールと検証方法を決める（この動画の内容）
② 公開前の最終チェック
媒体によっては人間の承認を挟む

正直に言うと：「100%放置」にしない場所もある

  • YouTube本体への投稿は手動 — 自動投稿はBANリスクがあるため、生成＋保存までで止める
  • Xの公開ボタンも人間 — 下書きまで自動。炎上リスクは人間が最終判断
  • つまり「制作は全自動、公開の責任だけ人間」が現実的な最適解

ここを雑にすると事故ります。安全装置はChapter 4の最後で解説

CHAPTER 4

実例：永久機関の作り方

全体のファイル構造

~/ClaudeCode/ ├── CLAUDE.md ← AIの行動ルール（憲法） ├── .claude/skills/ ← 媒体別の投稿スキル（部品） │ ├── x-post/ ← X投稿文の作り方 │ ├── note-post/ ← note記事の作り方 │ └── tips-post/ ← Tips記事の作り方 ├── memory/ ← 文体・失敗パターンの記憶 └── output/ ← 生成物の置き場

ループ本体は短い。賢さの本体は CLAUDE.md と skills と memory に分散している

CLAUDE.md ＝ ループの憲法

毎回のプロンプトに書かずとも、AIが常に守るルールをここに書く。例えば：

  • 媒体ごとの文体規定（noteは装飾区切り、Xは口語キャッチ…）
  • 禁止事項（YouTube自動投稿禁止／公開前に必ず確認…）
  • 固有名詞・数字はファクトチェック必須
  • 失敗したら memory に記録（同じミスを2度しない）
ルールを足すほど、放置できる範囲が広がる

スキル ＝ ループが使う部品

媒体ごとの「作り方の手順書」をスキル（スラッシュコマンド）にしておくと、ループはそれを呼ぶだけで済む。

/x-post /note-post /tips-post /srt（字幕） /cut（無音カット）

一度スキル化した作業は、二度と手順を説明しなくていい。
ループの品質 ＝ スキルの品質。

実際の /loop プロンプト

claude — ~/ClaudeCode
> /loop
今日公開したYouTube動画の文字起こしを素材に、
X・note・Tips・Threads向けの投稿を順番に作成して。

各媒体は対応するスキル（/x-post /note-post /tips-post）を使うこと。
完成したら output/ に保存し、検証チェックリストを通すこと。
全媒体が終わったら、成果物一覧のレポートを作って終了。

これだけ。あとは何時間でも勝手に回る。「何を・どの部品で・どう確認して・いつ終わるか」が全部入っているのがポイント

ゴール設計：ダメな例 と 良い例

ダメなゴール

「SNSがんばって」

  • 何をすればいいか曖昧
  • 終わりがない → 暴走する
  • できたか確認できない
良いゴール

「note記事1本を下書き保存まで。
3,000字以上・タイトル3案・ファクトチェック済み」

  • 具体的・検証可能・終了条件あり

検証方法の設計が、品質を決める

「できたかどうかをAI自身が確かめる方法」をループに書いておく：

  • 文字数・構成がフォーマット通りか
  • 固有名詞・数字・日付のファクトチェック
  • URLが実際に開けるか
  • 下書きが本当に保存されたか（画面で確認）
Claude Code の生みの親いわく、自己検証を入れると成果物の品質は2〜3倍になる

実際に投稿されたもの：X

ここに実際のX投稿の
スクリーンショット
  • 冒頭キャッチ＋動画URL＋学べること＋タイムスタンプの定型
  • 文体は memory のフォーマット規定から自動適用
  • 人間がやったのは「公開ボタンを押す」だけ

実際に投稿されたもの：note ／ Tips

ここにnote記事の
スクリーンショット

note：3,000字＋装飾区切り＋下書き保存まで自動

ここにTips記事の
スクリーンショット

Tips：5,000字記事を投稿まで自動

暴走させないための安全装置

  1. 上限を決める
    回数・時間・コストの上限をループに書いておく（無限は事故のもと）
  2. 「公開」は人間の承認制にする
    生成・下書きまで自動、世に出る瞬間だけ人間が押す
  3. 止め方を知っておく
    Ctrl+C で即停止。「◯◯になったら止まれ」という停止条件も書ける

媒体別：ゴールと自動化レベル早見表

媒体ループのゴール検証公開
YouTube台本・字幕・概要欄の生成字幕同期・誤字手動（BAN対策）
X投稿文＋タイムスタンプフォーマット・URL下書き→人間承認
Instagramリール台本・キャプション冒頭3秒フック半自動
TikTokショート台本完了率を意識した構成半自動
note記事3,000字＋タイトル3案文体・ファクト下書き保存まで自動
Tips記事5,000字固定構成・体裁投稿まで自動
Threads短文・連投セット文体・分割半自動

よくある失敗 3つ

  1. ゴールが曖昧で暴走する
    →「終了条件」を必ず書く。良いゴール＝具体的・検証可能・終わりがある
  2. 検証なしで低品質を量産する
    → 量が出るぶん、検証ステップがないと「ゴミの自動生成機」になる
  3. 全自動に欲張って事故る
    → 公開の最終判断は人間。媒体の規約と BAN リスクは必ず確認
